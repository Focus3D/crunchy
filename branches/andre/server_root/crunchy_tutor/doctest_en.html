People that know about these things apparently say that you should test your code. One method that is used in Python is known as DocTest. I am sure you know about DocTests; if not, you probably should make a note to read a real Python tutorial.

Jeff Elkner wrote on edu-sig that Python tutorials should perhaps be based around the use of DocTest. Here's an example of what he meant: try your hand at writing just enough code to make the following DocTest pass.

>>> double(21)
42
>>> double('Ha! ')
'Ha! Ha! '

After making a few (ahem) intentional mistakes to verify how friendly the error messages are, you will most certainly write the correct code to see what kind of congratulatory message you are going to get. Afterwards try the next, slightly more difficult example.

>>> p = Teacher('Jeff')
>>> p.name
'Jeff'
>>> p.addFriend('André')  # Crunchy [doctest?] bug: can't deal with non-ascii as in André
>>> p.friends
['André']

If you are using Crunchy, you realise that you can simply enter your code in the editor above, press the button, and see what happens.

If you are not using Crunchy, and you still want to try your hand at this DocTest business, you have to:

  • copy the above test in a file (which can be the same as your program);
  • edit your program to add the desired code;
  • look up the syntax to call up the appropriate method in the doctest module and feel like you are wasting too much time on something that should be much easier if life were fair.

Crunchy's ultimate goal is to make your life seem more fair.

Advanced stuff, for tutorial writers

<pre title="doctest [no_style] [linenumber]"></pre> adds a doctest to the page; the text between the pre tags is always placed before the textarea and used as the doctest.

The output from the doctest result is further processed by Crunchy to make error messages a bit friendlier to beginners.
