People that know about these things apparently say that you should test your code. One method that is used in Python is known as DocTest. I am sure you know about DocTests; if not, you probably shouldn't be reading this, but rather read a real Python tutorial.

Jeff Elkner wrote on edu-sig that Python tutorials should perhaps be based around the use of DocTest. Here's an example of what he meant: try your hand at writing just enough code to make the following DocTest pass.

>>> p = Animal()
>>> p.name
''
>>> p.friends
[]

After making a few (aham) intentional mistakes to verify how friendly the error messages are, you will most certainly write the correct code to see what kind of congratulatory message you are going to get. Afterwards try the next, slightly more difficult example.

>>> p = Animal('Pooh')
>>> p.name
'Pooh'
>>> p.addFriend('Piglet')
>>> p.friends
['Piglet']

If you are using Crunchy Frog, you realise that you can simply enter your code in the "box" above, press the button, and see what happens.

If you are not using Crunchy Frog, and you still want to try your hand at this DocTest business, you have to:

  • copy the above test in a file (which can be the same as your program);
  • edit your program to add the desired code;
  • look up the syntax to call up the appropriate method in the doctest module and feel like you are wasting too much time on something that should be much easier if life were fair.

Crunchy Frog's ultimate goal is to make your life seem more fair.

If you prefer things that are not fair, use this link to go to the canvas tutorial; however, you won't be able to see anything interesting if you are not using Crunchy Frog.
