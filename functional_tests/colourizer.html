Bugs

  • An interpreter session that ends with a comment, as in: 
    >>> # this last statement is a comment
    is currently processed by crunchy ("colourized") in such a way that it results in non W3C compliant code - and Crunchy (i.e. the elementtree handler) chokes on it, and displays a blank page. This should be fixed.
  • Code inside a <pre> that contains html markup is not processed properly. The suggested behaviour at this point would be to [1.] convert all <br> or <br/> into \n and [2.] to strip the remaining html markup. Here's an actual example using a <br/> rather than an "\n" character to separate lines of code. The actual code we use inside a <pre> element is:
    print "Hello world!"<br/>print "Crunchy rocks!"
    Here is how this example is processed by Crunchy: 
    print "Hello world!"
    print "Crunchy rocks!"
    Note how everything after the <br/> disappears.
  • Comments and linenumbers apparently don't mix properly [overlapping <span> sometimes result in a small interpreter prompt (i.e. in the same font size as the linenumber)]. Compare the size of the interpreter prompts on the first and second lines for example. 
    >>> # Comment on the first line.
>>> for i in range(10):
...    print i,
>>> print "Hello World"
>>> # Another comment.
>>> print "Hello World"
  • There is a bug with an editor (with line numbers) where a single empty line appears at the beginning, and a space is inserted before the first line of the output using Python 2.5 and Mac OS. 
    for i in range(5):
    print i,
print "Hello World"
    This last bug will need to be investigated further but is certainly not a show-stopper at the moment.
