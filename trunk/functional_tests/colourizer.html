In versions up to 0.8.1, comments and linenumbers apparently did not mix properly [overlapping <span> sometimes result in a small interpreter prompt (i.e. in the same font size as the linenumber)]. In the following example, all interpreter prompts should appear the same. In particular, compare the interpreter prompts on the first and second lines which used not to be the same font size. 

>>> # Comment on the first line.
>>> for i in range(10):
...    print i,
>>> print "Hello World"
>>> # Another comment.
>>> print "Hello World"

Bugs

  • An interpreter session that ends with a comment, as in: 
    >>> # this last statement is a comment
    used to be processed by crunchy ("colourized") in such a way that it resulted in non W3C compliant code - and Crunchy (i.e. the elementtree handler) chokeed on it, and displayed a blank page. This has been fixed ... but, somehow, this text that appears outside of the "pre" is taken to be part of it.
  • Code inside a <pre> that contains html markup is not processed properly. The suggested behaviour at this point would be to [1.] convert all <br> or <br/> into \n and [2.] to strip the remaining html markup. Here's an actual example using a <br/> rather than an "\n" character to separate lines of code. The actual code we use inside a <pre> element is:
    print "Hello world!"<br/>print "Crunchy rocks!"
    Here is how this example is processed by Crunchy: 
    print "Hello world!"
    print "Crunchy rocks!"
    Note how the line break is lost ... and how this line is actually styled as code!?
  • The following is another "bare pre" (<pre>) containing some marked up code, which is technically formatted properly. 
    >>> for i in range(3):
...     print "Hello world!"
... 
"Hello world!"
"Hello world!"
"Hello world!"
    More of the same (2 lines of code; a simple for loop): 
    for i in range(3):
    print "Hello world!"
